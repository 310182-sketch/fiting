## Plan: 本地健身 PWA 最終 MVP 規劃

MVP 是一個本地優先的健身紀錄 PWA：先選「模板」（一組固定動作清單），進入訓練流程時「先熱身再正式」，可記錄力量與有氧；所有資料存在 IndexedDB。Dashboard 以「日曆週（週一～週日）」為單位，統計每週訓練天數、總時間、總組數、總重量（排除熱身組），並以 kg/lb 全域切換＋與上週比較的 ±% 數字呈現進步狀況，小數與顏色規則都事先定義好。

### Steps
1. 完整定義資料模型：在規劃中敲定 `Exercise`（含 type=strength/cardio）、`WorkoutTemplate`（id, name, exerciseIds[]）、`Workout`（含 templateId）、`Set`（weight, reps, duration, distance, unit, isWarmup 等）欄位與關係。
2. 規範單位與小數處理：在計畫裡寫明 DB 一律以 kg 儲存重量，提供全域 `weightUnit`（kg/lb）切換；定義 kg↔lb 轉換公式與顯示規則（例如四捨五入到固定小數或 0.5 單位），確保統計與 UI 一致。
3. 設計訓練 UX（先熱身再正式）：在使用流程中固定為「選模板 → 建立 Workout → 對每個動作先新增熱身組（預設 isWarmup=true）再新增正式組（isWarmup=false）」，並描述在 UI 中如何標記與切換熱身／正式。
4. 定義週統計與同比邏輯：在統計設計裡寫清楚「以週一～週日為一週，對所有非熱身力量組計算每週總組數與總重量 Σ(weight×reps)，再計算本週/上週各指標的 ±% 變化，並處理上週=0 的情況」。
5. 規劃 Dashboard 呈現與顏色規則：在首頁設計中定義一個「本週卡片」，顯示訓練天數、總時間、總組數（含有氧、排除熱身）、總重量（正式力量組），並指定：當 ±% 變化超過門檻（例如 |Δ| ≥ 2%）時以顏色強調上升/下降，其餘以中性顏色和標籤「持平」呈現。
6. 整合更新規劃檔：將資料模型、訓練流程、單位策略、週統計與視覺呈現等條文，重新整理後寫入 `plan-fiting.prompt.md`，作為後續實作的權威藍本。

### Further Considerations
1. 熱身預設行為：在文檔中是否要明確寫成「每個新動作的第一組預設為熱身組，其後新增組預設為正式組」，讓實作和操作都更直覺？
2. 小數顯示細節：kg/lb 顯示是統一到 1 位小數、0.5 間距，還是根據單位有不同規則（例如 kg 到 0.5、lb 到整數）？
3. 顏色與門檻細節：±% 的門檻（例如 2% 或 5%）與顏色（綠/紅/灰）是否要在計畫中標準化命名，方便之後樣式與可視化統一？
